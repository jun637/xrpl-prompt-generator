## 2. 왜 마스터 프롬프트 → Generator 인가 (가장 중요한 설계 원칙)

KFIP 참가자들에게 마스터 프롬프트를 그대로 던졌을 때 가장 큰 마찰은 **"빈 칸을 직접 채우기 귀찮다"** 였다. 본 앱은 이 문제를 푸는 데 모든 디자인 결정을 정렬한다.

따라서 좌측 폼의 **디폴트는 "선택"** 이고, "직접 입력" 은 fallback 이다.

- 모든 질문에는 **미리 제시된 보기 옵션**(라디오·체크박스·드롭다운·칩)이 먼저 노출된다
- 사용자가 보기 중에서 고르면 그 라벨이 그대로 우측 프롬프트의 답변으로 박힌다
- 보기에 원하는 게 없을 때만 "기타 / 직접 입력" 옵션을 눌러 텍스트를 친다
- 1줄로 끝나는 질문은 무조건 선택형, 여러 줄이 필요한 질문(MVP 범위·고민되는 부분 등)만 textarea

→ 결과적으로 **사용자는 "고민 → 타이핑" 이 아니라 "고민 → 클릭" 으로 마스터 프롬프트를 완성**한다. 이게 generator 화의 핵심 가치이며, 단순한 폼 래퍼와 구분되는 지점이다.

## 3. 콘셉트 (참고 캡쳐 4장 설명)

`references/pg1.png ~ pg4.png` 는 ai-narrative.io 의 PPT 프롬프트 생성기 UI 캡쳐. 우리가 만들 앱의 **레이아웃·인터랙션 패턴 레퍼런스**이지, 디자인 복제 대상이 아님.

핵심 패턴 (캡쳐에서 가져올 것):
- **2-컬럼 레이아웃**: 좌측 폼 / 우측 sticky 프리뷰 (생성되는 프롬프트)
- **섹션별 아코디언/스텝**: `01 …`, `02 …`, `03 …` 처럼 번호로 나누어진 섹션
- **선택지 카드/칩 우선 UI**: 라디오·체크박스·세그먼트가 라벨 카드 형태로 노출 ← **이 패턴이 우리 앱의 본질**
- **우측 프리뷰는 라이브 업데이트**: 좌측 폼이 바뀔 때마다 프롬프트가 즉시 재생성
- **상단 우측 복사 버튼**: 한 번에 전체 프롬프트 복사

캡쳐의 4 섹션 구성(`01 메시지 / 02 디자인 / 03 슬라이드별 내용 / 04 고급 설정`)은 PPT 생성기의 도메인 컨벤션이므로 그대로 가져오지 말 것. 우리 도메인(XRPL 제품 설계 프롬프트)에 맞춰 재구성한다.

## 4. 핵심 콘텐츠 — XRPL 마스터 프롬프트 본문

KFIP 참가자에게 공유되어 반응이 좋았던 마스터 프롬프트가 **앱의 출력 본체**이다. **두 가지 모드**가 있고, 모드별로 별도의 마스터 프롬프트를 사용한다. 사용자의 폼 입력값은 마스터 프롬프트 안의 빈 라인 (`- 우리 팀은 어떤 팀인지:`, `- 현재 주요 고객은 누구인지:` 등) 뒤에 그대로 채워져 출력된다.

### 4.1 인트로 문구 (앱 헤더/설명에 사용)

이 도구는 XRPL 기반 제품 구조를 빠르게 구체화하기 위한 실무용 마스터 프롬프트를 생성합니다.

- **의사결정권자**는 아이디어나 파트너십 제안이 실제 비즈니스 가치와 연결되는지 빠르게 검토할 수 있습니다.
- **기획자**는 막연한 아이디어를 사용자 흐름, 핵심 기능, 온체인 트랜잭션 구조, MVP 범위로 정리할 수 있습니다.
- **개발자**는 XRPL 적용 지점, 온체인/오프체인 역할 분리, 시스템 아키텍처, 구현 우선순위를 빠르게 파악할 수 있습니다.

핵심은 "XRPL을 어디에 붙일까?"가 아니라, **사용자가 어떤 행동을 반복하고, 그 행동이 왜 온체인 트랜잭션으로 기록될 가치가 있는가?** 를 찾는 것입니다.

### 5.2 핵심 인터랙션

**입력 패턴 (가장 중요)**:
- **싱글 선택 (라디오 카드)**: 4~7개 보기 + 마지막에 "기타 (직접 입력)" 카드. 기타 선택 시 카드 아래에 인라인 텍스트 입력 노출. 답이 하나만 박혀야 하는 질문(예: "한국 중심인지 글로벌인지")에 사용.
- **멀티 선택 (체크박스 칩)**: 보기들을 칩 형태로 펼쳐 다중 선택. 선택된 칩만 라벨이 프롬프트에 콤마로 합쳐져 박힘. "기타" 칩 별도. 사례·기능 후보 나열 질문에 사용.
- **세그먼트 컨트롤**: 보기가 2~3개인 짧은 질문(B2B/B2C/B2B2C 등)은 가로 세그먼트 토글.
- **드롭다운**: 보기가 8개 이상이거나 시각적으로 펼치면 산만한 질문에 한해 사용. 디폴트는 아님.
- **textarea (직접 입력)**: 정말 자유 서술이 필요한 질문에만 한정 사용 — 예) "꼭 검증하고 싶은 가설", "특히 고민되는 XRPL 적용 포인트", "한 줄 소개". 그 외 모든 질문은 선택형이 디폴트.
- **혼합형 (선택 + 자유 추가)**: 멀티 선택 + "추가로 적고 싶은 게 있으면" textarea 한 줄. 사용자가 옵션으로 80% 채우고 20% 만 직접 쓰는 흐름.

**전체 흐름**:
- **모드 토글**: 상단 큰 세그먼트(2개 버튼). 전환 시 폼 카피와 우측 마스터 프롬프트가 함께 바뀜. 입력값은 가능한 한 보존(공통 키만 살리고 나머지는 비움).
- **섹션 헤더**: `01 Who`, `02 What` … 처럼 번호 + 라벨. 아코디언 접기/펼치기 옵션(기본 펼침).
- **우측 프리뷰**: 좌측 onChange 마다 디바운스 100~200ms 후 재계산. `{{var}}` 치환은 단순 문자열 replace 면 충분.
- **빈 입력 표시**: 채워지지 않은 자리에는 옅은 회색 텍스트로 "여기에 답변" 같은 placeholder 유지 → 사용자가 어디까지 채웠는지 한 눈에 보이도록.
- **복사 버튼**: 클립보드 복사 후 "복사됨!" 토스트 1.5s.
- **다운로드 버튼**: `.txt` 또는 `.md` 파일로 저장. 파일명 `xrpl-prompt-{YYYYMMDD-HHmm}.txt`.
- **로컬 저장**: `localStorage` 로 입력값 자동 저장(키별). 새로고침해도 유지. "초기화" 버튼 별도.
- **공유 링크 (v1 이후)**: 입력값을 querystring 또는 base64 로 인코딩해 URL 공유. v0 에서는 보류.

## 6. 기술 스택

  - SSL 자동 발급
- **저장소**: GitHub. 리포 이름 잠정 `xrpl-prompt-generator` (Jun 과 합의 후 확정)
